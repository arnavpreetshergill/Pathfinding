ABSTRACT

Navigation and route planning have become essential capabilities in modern geographic information systems. While cloud-based services such as Google Maps and Mapbox Directions have made online routing widely accessible, a significant class of operational environments exists where Internet connectivity is unavailable, unreliable, or undesirable. Remote field operations, disaster response zones, rural areas with poor coverage, conservation research stations, humanitarian aid corridors, mining and exploration sites, and environments with strict data privacy requirements all present scenarios where cloud-dependent mapping solutions cannot function.

This project presents the design and implementation of a fully offline, self-contained routing engine capable of computing shortest vehicular paths over real-world road networks without any external network dependency. The system ingests OpenStreetMap (OSM) data, transforms it into a routable graph within a PostgreSQL database augmented with PostGIS spatial extensions and the pgRouting library, and exposes pathfinding capabilities through a Spring Boot REST API that executes the A* (A-Star) graph search algorithm with Euclidean distance heuristics.

The frontend is a cross-platform desktop application built with Angular 18 and Electron, featuring an interactive Leaflet.js map interface that renders pre-downloaded raster tiles for fully offline cartographic visualisation. The system supports coordinate snapping via k-Nearest Neighbour (kNN) spatial queries, same-edge sub-segment extraction, dynamic road network rendering in the viewport, and can be packaged as a portable executable for both Windows and Linux platforms.

The architecture achieves complete offline compliance while maintaining sub-second routing performance for city-scale queries over a road network containing millions of edges, demonstrating the viability of open-source geospatial tools for production-grade offline navigation.

Keywords: Offline Routing, A* Algorithm, pgRouting, PostGIS, OpenStreetMap, Graph Theory, Spatial Databases, Electron, Angular, Leaflet.js


1. INTRODUCTION

1.1. PROBLEM STATEMENT

Navigation and route planning have become ubiquitous in daily life through cloud-based services such as Google Maps, Apple Maps, and Mapbox. These platforms rely on continuous Internet connectivity to fetch map tiles, compute routes on remote servers, and deliver real-time traffic-aware directions. However, a significant class of operational environments exists where this connectivity assumption fundamentally fails.

Disaster Response Zones: Natural disasters such as earthquakes, floods, hurricanes, and cyclones routinely destroy cellular and wired communication infrastructure. First responders and humanitarian aid workers require immediate route planning capabilities in areas where network services have collapsed. In the critical first 72 hours after a major disaster, communication infrastructure is often the last thing restored, yet route planning is among the first things needed. Emergency shelters must be located, supply routes established, and evacuation paths identified — all while working with no cloud connectivity whatsoever.

Remote and Underserved Regions: Large portions of rural territories, mountainous terrain, dense forest areas, and geographically isolated regions lack reliable cellular coverage. Vehicles operating in these regions — whether for agricultural logistics, forestry management, mining operations, or rural healthcare delivery — cannot depend on cloud-based navigation. Even when cellular coverage nominally exists, signal strength in valleys, dense canopy areas, and mountainous terrain is often insufficient for the sustained data transfer required by streaming map tile services.

Data Privacy and Sovereignty Constraints: In certain organisational and institutional contexts, transmitting geographic coordinates and routing queries to third-party cloud servers poses data security and privacy risks. Sensitive location data — such as the movements of valuable cargo, the locations of critical infrastructure, or the travel patterns of key personnel — should remain within local infrastructure. Regulations such as GDPR in Europe and various national data localisation laws increasingly require that geographic data processing occur within specified jurisdictions.

Bandwidth-Constrained Environments: Even where intermittent connectivity exists, streaming high-resolution map tiles and executing server-side routing queries consumes significant bandwidth that may be reserved for other critical communications. A routing system that operates entirely locally eliminates this bandwidth competition entirely.

Isolated Network Environments: Certain facilities, research stations, industrial complexes, and operational environments run on physically isolated (air-gapped) networks with no Internet access by design, yet still require route planning capabilities for logistics and transportation planning.

Cost-Sensitive Deployments: Cloud-based mapping services often come with usage-based pricing that can become prohibitive for organisations with high query volumes or limited budgets. An offline routing system built entirely on open-source components eliminates these ongoing costs while providing comparable core functionality.

Reliability-Critical Operations: Any system that depends on external cloud services inherits the reliability characteristics of those services and the network infrastructure connecting to them. For operations where routing must be available with near-100% uptime, the additional points of failure introduced by cloud dependency are unacceptable.

The core problem this project addresses is: How can a fully self-contained vehicular routing system be designed and deployed on standard hardware that computes shortest paths over real-world road networks without any external network dependency, while maintaining performance adequate for interactive use?

This is not merely a matter of caching map tiles. The system must perform the complete routing computation pipeline locally: storing the road network graph in a spatial database, executing graph search algorithms with spatial heuristics, snapping arbitrary user-provided coordinates to the nearest road segment, handling edge cases such as same-edge routes, dynamically rendering the road network based on the user's current viewport, and presenting results on an interactive map — all without a single outbound network request.

2. LITERATURE SURVEY

2.1. LITERARY SURVEY

This section reviews the existing body of work in graph-based pathfinding, spatial databases, offline navigation, and the specific open-source technologies employed in this project.


2.1.1. Graph-Based Shortest Path Algorithms

The computation of shortest paths in weighted graphs is one of the most extensively studied problems in computer science. The foundational algorithms relevant to road network routing are discussed below.

Dijkstra's Algorithm (1959): Edsger W. Dijkstra introduced his single-source shortest path algorithm in 1959 [1]. The algorithm operates on weighted graphs with non-negative edge weights and guarantees finding the optimal shortest path from a source vertex to all other vertices. Using a min-priority queue (typically a binary heap), Dijkstra's algorithm achieves a time complexity of O((V + E) log V), where V is the number of vertices and E is the number of edges. For road networks, Dijkstra's algorithm explores vertices in concentric circles outward from the source without any knowledge of where the target lies, which means it examines many vertices that are geographically irrelevant to the optimal path. On a large-scale road network with millions of vertices, this uninformed exploration results in unacceptable computation times for interactive use. Despite this limitation, Dijkstra's algorithm remains important as the foundation upon which more advanced algorithms are built.

A* Algorithm (Hart, Nilsson, and Raphael, 1968): The A* algorithm, introduced by Hart, Nilsson, and Raphael in 1968 [2], extends Dijkstra's algorithm by incorporating a heuristic function h(n) that estimates the cost from any vertex n to the goal vertex. The total evaluation function is f(n) = g(n) + h(n), where g(n) is the actual cost from the start vertex to n, h(n) is the heuristic estimate from n to the goal, and f(n) is the estimated total cost. The algorithm is guaranteed to find the optimal shortest path if the heuristic is admissible (never overestimates the true cost) and consistent (satisfies the triangle inequality). For road networks, the Euclidean straight-line distance serves as a natural admissible heuristic, since no road path can be shorter than the direct straight-line distance. The key advantage of A* over Dijkstra's is its directed search: by prioritising vertices that are both close to the source and close to the goal, A* explores far fewer vertices. Empirical studies on road networks show that A* typically explores 50–80% fewer vertices than Dijkstra's algorithm [5].

Contraction Hierarchies (Geisberger et al., 2008): Contraction Hierarchies (CH), introduced by Geisberger et al. in 2008 [3], represent a major advance in practical road network routing. The technique involves a preprocessing phase that contracts vertices in order of importance, creating shortcut edges that bypass less important vertices. During preprocessing, vertices are ordered by heuristics such as edge difference and search space size, then contracted from least to most important. Contracting a vertex involves removing it and adding shortcut edges between its neighbours where the shortest path passed through the contracted vertex. After preprocessing, queries are answered by a bidirectional Dijkstra search on the augmented graph that only relaxes edges going upward in the hierarchy, achieving query times in the microsecond range. While CH provides dramatically faster query times than A*, it requires a substantial preprocessing step (hours for large networks) and significant additional storage for shortcut edges. The pgRouting library does not natively support CH, which motivated the selection of A* as the primary algorithm for this project.

Bidirectional Search Variants: Bidirectional Dijkstra and bidirectional A* algorithms initiate searches simultaneously from the source and the target, terminating when the two search frontiers meet. These approaches can reduce the explored search space by approximately 30–50% compared to their unidirectional counterparts. pgRouting supports bidirectional Dijkstra (pgr_bdDijkstra) and bidirectional A* (pgr_bdAstar), though the unidirectional A* was selected for this project due to its simplicity and adequate performance characteristics.


2.1.2. Spatial Database Systems and PostGIS

Traditional relational databases are optimised for one-dimensional data (integers, strings, dates) and cannot efficiently handle spatial queries such as "find the nearest road to this coordinate" or "retrieve all roads within this bounding box." Spatial database extensions address this limitation through specialised data types, indexing structures, and query operators.

PostGIS [7] is the premier open-source spatial extension for PostgreSQL. It adds support for geographic objects (points, lines, polygons) as first-class database types and provides over 300 spatial functions for distance calculations, geometric transformations, coordinate system conversions, and spatial relationship testing. Key PostGIS functions used in this project include ST_MakePoint (constructs point geometry), ST_SetSRID (assigns WGS 84 coordinate reference system), ST_Length (computes geodesic length in metres), ST_ClosestPoint (finds nearest point on a linestring), ST_AsGeoJSON (serialises geometry to GeoJSON), ST_MakeEnvelope (constructs bounding box), ST_LineSubstring (extracts portion of a linestring), and ST_LineLocatePoint (returns fractional position along a linestring).

PostGIS spatial queries achieve their performance through the Generalised Search Tree (GiST) indexing framework [6]. GiST indexes on geometry columns build R-tree structures that partition space into nested bounding rectangles. This enables the k-Nearest Neighbour (kNN) distance operator (<->) to perform O(log N) nearest-neighbour lookups instead of O(N) sequential scans, which is critical for the coordinate snapping operation in the routing service.


2.1.3. pgRouting Library

pgRouting [8] extends PostgreSQL/PostGIS with geospatial routing functionality. It implements over 30 graph algorithms as SQL-callable functions, including pgr_dijkstra (Dijkstra's shortest path), pgr_aStar (A* with heuristic guidance), pgr_createTopology (automatic topology builder), pgr_ksp (K-shortest paths), and pgr_TSP (Travelling Salesman Problem approximation).

The pgr_aStar function requires edge data with the following columns: id, source, target, cost, reverse_cost, and the heuristic coordinate columns x1, y1, x2, y2. In this implementation, all of these except source and target are computed on-the-fly from the stored geometry column, reducing storage requirements.

The distinguishing characteristic of pgRouting compared to standalone routing engines (OSRM, GraphHopper) is that it operates directly on data stored in PostgreSQL tables. This means the routing graph is always in sync with the database and can be dynamically modified through standard SQL operations, enabling standard SQL-based administration, backup, and replication without maintaining a separate routing daemon.


2.1.4. OpenStreetMap Data and Processing

OpenStreetMap (OSM) [9] is a collaborative, open-data mapping project that provides free geographic data for the entire world. The road network in OSM contains millions of road segments tagged with classification attributes (highway=motorway, highway=primary, etc.) and directional metadata (oneway=yes/no).

Two primary tools exist for importing OSM data into PostgreSQL. osm2pgrouting directly imports OSM data into a pgRouting-compatible schema with source/target topology, suitable for straightforward routing setups. osm2pgsql is a general-purpose OSM importer that creates the planet_osm_line table, requiring additional SQL transformation to extract drivable roads and build routing topology. This project supports both import pathways, with detailed SQL scripts for the osm2pgsql transformation workflow.


2.1.5. Offline Navigation Systems

The concept of offline navigation is well-established in several existing products. OsmAnd [10] is an open-source mobile application that provides offline routing using pre-computed vector maps and a custom routing engine optimised for mobile hardware. HERE WeGo is a commercial offline navigation application that downloads regional map packages for offline use. OSRM (Open Source Routing Machine) [11] is a high-performance C++ routing engine that pre-processes OSM data using Contraction Hierarchies for sub-millisecond query times, but operates as a separate server process and does not integrate with PostgreSQL. GraphHopper [12] is a Java-based routing engine supporting multiple algorithms including CH and Landmark-based routing, but similarly maintains its own data structures separate from the database.

The distinguishing characteristic of the approach used in this project is the use of PostgreSQL as both the data store and the computation engine via pgRouting, eliminating the need for a separate routing daemon and enabling standard SQL-based administration.


2.1.6. Electron Desktop Application Framework

Electron [13] enables the development of cross-platform desktop applications using web technologies (HTML, CSS, JavaScript). It embeds Chromium for rendering and Node.js for system access. Key architectural decisions in Electron applications include context isolation (the renderer process runs in a sandboxed environment with no direct access to Node.js APIs), preload scripts (a secure bridge between the renderer and main processes using contextBridge.exposeInMainWorld()), and packaging (tools like electron-builder produce platform-specific installers and portable executables).


2.2. THEORETICAL UNDERSTANDING

This section provides the mathematical and algorithmic foundations underlying the routing engine's core operations.


2.2.1. Graph Theory Foundations

A road network is naturally modelled as a weighted graph G = (V, E, w) where V = {v1, v2, ..., vn} is the set of vertices (road intersections), E is a subset of V x V representing the set of edges (road segments connecting intersections), and w: E -> R+ is the weight function assigning a non-negative cost (road length in metres) to each edge.

In our implementation, each row in the "PanIndiaRoad" table represents an edge with source and target vertex IDs. Each row in the "PanIndiaRoad_vertices_pgr" table represents a vertex with its geographic position. The weight w(e) is computed as ST_Length(geom::geography), which returns the geodesic distance in metres along the road segment's geometry.

Road networks exhibit several structural properties that influence algorithm performance. They are approximately planar (roads generally do not cross without intersections), have low vertex degree (most intersections connect 2–4 roads), exhibit spatial locality (shortest paths tend to use geographically local edges), and possess a natural hierarchical structure (local streets, collector roads, arterials, highways, motorways).


2.2.2. The A* Algorithm in Detail

The A* algorithm maintains two data structures: an open set (priority queue) containing vertices to be evaluated, ordered by f(n) = g(n) + h(n), and a closed set containing vertices already fully evaluated.

Algorithm: A* Shortest Path
Input: Graph G = (V, E, w), start vertex s, goal vertex t, heuristic h
Output: Shortest path from s to t

Step 1: Set g(s) = 0, f(s) = h(s, t). Initialise openSet = {s}, closedSet = empty. Set parent(v) = null for all v.

Step 2: While openSet is not empty, extract vertex n with minimum f(n).

Step 3: If n = t, reconstruct and return the path by following parent pointers from t back to s.

Step 4: Move n from openSet to closedSet.

Step 5: For each neighbour m of n connected by edge (n, m): if m is in closedSet, skip it. Compute tentative_g = g(n) + w(n, m). If m is not in openSet or tentative_g < g(m), update parent(m) = n, g(m) = tentative_g, f(m) = g(m) + h(m, t), and add m to openSet if not already present.

Step 6: If openSet becomes empty without finding t, return "No path exists."

For geospatial routing, the Euclidean distance heuristic is defined as h(n, t) = sqrt((xn - xt)^2 + (yn - yt)^2), where (xn, yn) and (xt, yt) are the geographic coordinates of vertex n and the target vertex t. This heuristic is admissible because the straight-line distance is always less than or equal to any road path connecting the two points.

In the pgRouting implementation, the heuristic coordinates are provided through x1, y1 (start point of edge) and x2, y2 (end point of edge) columns, computed on-the-fly from the geometry using ST_X(ST_StartPoint(geom)), ST_Y(ST_StartPoint(geom)), ST_X(ST_EndPoint(geom)), and ST_Y(ST_EndPoint(geom)).

Optimality: If h(n) is admissible and consistent, then A* is guaranteed to find the shortest path from s to t if one exists.

Time Complexity: In the worst case, A* has the same complexity as Dijkstra's algorithm: O((V + E) log V). However, with a good heuristic, the number of expanded vertices is significantly reduced. For road networks with approximately planar graph structure, empirical performance is often close to O(sqrt(V)) per query [4].

Space Complexity: O(V) for the open and closed sets, plus O(E) for the graph representation (stored in the database).


2.2.3. Coordinate Snapping via k-Nearest Neighbour (kNN)

When a user clicks on the map, the clicked coordinate will almost never coincide exactly with a vertex in the road network. The system must snap this coordinate to the nearest road segment and identify the closest network vertex.

The snapping operation uses the PostGIS kNN distance operator (<->). First, the nearest road segment is found: e* = argmin over all edges e of d_geom(e.geom, point(lng, lat)). The GiST spatial index enables this operation in O(log N) time. Then, the exact snapped point on the edge is computed using ST_ClosestPoint, and the system determines whether the source or target vertex is closer to the snapped point. The fractional position along the edge is computed using ST_LineLocatePoint, which returns a value in [0, 1] representing where the snapped point falls along the linestring.


2.2.4. Same-Edge Sub-Segment Extraction

A special case arises when both the start and end coordinates snap to the same road edge. In this scenario, running a full A* search would be wasteful. Instead, the system extracts the sub-segment of the edge between the two fractional positions: sub_geom = ST_LineSubstring(e.geom, min(fs, fe), max(fs, fe)), where fs and fe are the fractional positions of the start and end snapped points respectively.


2.2.5. Bounding Box Spatial Queries

The dynamic road rendering feature fetches all road segments visible in the current map viewport. This uses the PostGIS envelope intersection operator (&&), which checks if the bounding boxes of two geometries overlap. The operator is an efficient index-supported operation that avoids exact geometric intersection tests.


2.2.6. Cost Function Design

The cost function for edge traversal uses geodesic distance: cost(e) = ST_Length(e.geom::geography). The ::geography cast is critical: it converts the geometry from Cartesian plane calculations to geodesic (great-circle) calculations on the WGS 84 ellipsoid, ensuring distance computations are accurate regardless of latitude. Without this cast, distances at high latitudes would be significantly underestimated due to the convergence of meridians.

In the current implementation, the graph is treated as undirected, with reverse_cost(e) = cost(e) for all edges. This simplification means one-way streets are not enforced. A future enhancement could set reverse_cost = -1 for edges marked with oneway=yes in the OSM data.


2.2.7. XYZ Tile Pyramid for Offline Map Rendering

The offline map rendering uses the Slippy Map Tilenames convention (XYZ tiles). Map imagery is pre-rendered at multiple zoom levels (z), with each zoom level containing 2^z x 2^z tiles. Each tile is addressed by three coordinates: z (zoom level, 0 to 14 in this system), x (horizontal tile index, 0 to 2^z - 1), and y (vertical tile index, 0 to 2^z - 1).

The tiles are stored as JPEG images in a directory hierarchy: offline-map-data/{z}/{x}/{y}.jpg. Leaflet.js requests tiles via this URL pattern, and the Electron/Angular application serves them from the local filesystem.


3. METHODOLOGY AND SOFTWARE REQUIREMENTS

3.1. METHODOLOGY

3.1.1. Development Approach

The project follows an iterative incremental development methodology, structured into six phases.

Phase 1 — Data Acquisition and Preparation: Download OpenStreetMap extracts from Geofabrik, import into PostgreSQL using osm2pgsql or osm2pgrouting, and transform raw data into a routable graph with proper topology, cost columns, and spatial indexes.

Phase 2 — Backend API Development: Implement the Spring Boot REST API with routing service (A* pathfinding, coordinate snapping) and health monitoring endpoints.

Phase 3 — Frontend Development: Build the Angular application with interactive Leaflet map, coordinate input controls, route visualisation, and dynamic road rendering.

Phase 4 — Desktop Integration: Wrap the Angular application in Electron for cross-platform desktop deployment with offline tile serving and native window management.

Phase 5 — Testing and Optimisation: Validate routing accuracy, measure performance, and optimise database queries and index configurations.

Phase 6 — Packaging and Deployment: Build distributable packages for Windows (.exe portable) and Linux (.tar.gz) using electron-builder.


3.1.2. System Architecture

The system follows a three-tier client-server architecture.

Presentation Tier: Angular 18 + Leaflet.js running inside an Electron Desktop Shell. The map interface loads tiles from a local tile store (XYZ JPEG tiles stored on disk).

Application Tier: Spring Boot 4.1 REST API running on Java 17 with embedded Tomcat. Contains the RoutingController, HealthController, RoutingService, and MaskedMessageConverter.

Data Tier: PostgreSQL 15 with PostGIS and pgRouting extensions. Contains the "PanIndiaRoad" edges table (gid, source, target, geom) and the "PanIndiaRoad_vertices_pgr" vertices table (id, the_geom).

Communication between tiers: The Presentation Tier communicates with the Application Tier via HTTP REST calls with JSON/GeoJSON payloads. The Application Tier communicates with the Data Tier via JDBC/SQL using parameterised queries.


3.1.3. Data Flow

The routing workflow proceeds as follows:

1. User clicks two points on the Leaflet map (or enters coordinates manually).
2. The Angular frontend sends an HTTP GET request to /api/routing/calculate with the four coordinate parameters.
3. The Spring Boot controller validates the coordinates and delegates to the RoutingService.
4. The RoutingService snaps both coordinates to the nearest road vertex using PostGIS kNN queries.
5. If both points snap to the same edge, a sub-segment is extracted using ST_LineSubstring.
6. Otherwise, pgr_aStar computes the shortest path between the two snapped vertices.
7. The resulting path geometries are serialised to GeoJSON and returned in the HTTP response.
8. The Angular frontend parses the GeoJSON segments and renders them as polylines on the Leaflet map.
9. The map auto-zooms to fit the computed route bounds.

5.3. Performance Analysis

Route computation times were measured for routes of varying distance across the road network:

Intra-city (within urban area, approx. 15 km, approx. 50 segments): less than 1 second.
Short inter-city (approx. 100 km, approx. 200 segments): 1–3 seconds.
Medium inter-city (approx. 270 km, approx. 500 segments): 2–5 seconds.
Long-distance (approx. 1,400 km, approx. 2,500 segments): 10–30 seconds.
Cross-country (approx. 2,200 km, approx. 4,000 segments): 30–60 seconds.

Note: Actual performance depends on hardware specifications, database configuration, and road network density.


5.4. Database Performance

The spatial indexing strategy proved critical for performance:

Coordinate Snapping (kNN): Without index, over 10 seconds. With GiST/B-Tree index, under 50 ms.
Bounding Box Road Query: Without index, over 5 seconds. With GiST/B-Tree index, under 200 ms.
A* Graph Traversal: Without index, minutes. With GiST/B-Tree index, seconds.

The three critical indexes are: GiST on geom (enables O(log N) spatial queries), B-Tree on source (accelerates graph traversal by pgRouting's edge loading), and B-Tree on target (complements the source index for bidirectional edge access).


5.5. Analysis of Design Trade-offs

A* vs. Contraction Hierarchies: A* was selected over CH due to pgRouting's native support and the absence of a preprocessing requirement. While CH would provide 100–1000x faster queries, the A* implementation achieves acceptable performance for the target use case.

On-the-fly Cost Computation: Computing cost, reverse_cost, and heuristic coordinates dynamically from geom reduces storage by approximately 40% compared to storing these as materialised columns. The computational overhead is negligible relative to the A* search time.

Undirected Graph Simplification: Treating all roads as bidirectional simplifies the data model but may produce routes that violate one-way street rules. For the current application scope focused on general route planning, this is an acceptable simplification.

JDBC over JPA/Hibernate: The backend uses raw JdbcTemplate queries instead of an ORM. This provides full control over the complex spatial SQL queries required by PostGIS and pgRouting, which would be difficult to express in JPQL or Criteria API.

GeoJSON as Wire Format: Returning route geometries as GeoJSON strings enables direct consumption by Leaflet's L.geoJSON() constructor without any client-side coordinate transformation.


6. CONCLUSION AND FUTURE PROSPECTS

6.1. Conclusion

This project successfully designed, implemented, and validated a fully offline vehicular routing engine capable of operating in environments where Internet connectivity is unavailable, unreliable, or undesirable. The system achieves its primary objectives:

Complete Offline Operation: The entire pipeline — from map tile rendering to route computation — functions without any network requests, meeting the strict requirements of connectivity-constrained environments.

Real-World Road Network: The system operates on actual OpenStreetMap road data containing millions of segments, demonstrating its applicability to real-world routing scenarios rather than synthetic benchmarks.

Optimal Pathfinding: The A* algorithm with Euclidean distance heuristics computes provably optimal shortest paths, with the spatial indexing infrastructure (GiST indexes, B-Tree indexes) enabling practical query times.

Interactive User Experience: The Angular/Electron frontend provides a responsive, interactive map interface with click-to-route, manual coordinate input, dynamic road rendering, and visual feedback (loading states, error alerts, auto-zoom).

Cross-Platform Deployment: The application packages as standalone executables for both Windows and Linux platforms via electron-builder.

Security Hardening: The credential masking system prevents accidental exposure of database passwords and tokens in log files, and the Electron configuration enforces context isolation, sandbox mode, and disabled node integration.

Modular Architecture: The clean three-tier separation (Angular frontend, Spring Boot API, PostgreSQL/PostGIS/pgRouting) enables independent upgrades and maintenance of each layer.

The project demonstrates that production-quality offline routing is achievable using entirely open-source tools (OpenStreetMap, PostgreSQL, PostGIS, pgRouting, Spring Boot, Angular, Electron, Leaflet.js), without dependence on proprietary mapping APIs or cloud infrastructure. This has significant implications for organisations operating under connectivity constraints, budgetary limitations, or data privacy requirements.


6.2. Limitations

The current implementation has the following limitations:

One-Way Street Enforcement: The graph is treated as undirected, meaning one-way streets, highway ramps, and divided highways are not accurately modelled.

No Turn Restrictions: Turn restrictions (e.g., "no left turn" at specific intersections) are not captured in the graph model.

Query Performance for Long Routes: Cross-country routes (over 1,000 km) may take 30–60 seconds, which exceeds the responsiveness threshold for time-critical scenarios.

Static Road Network: The road network reflects the state of OpenStreetMap at the time of data import. Road closures, construction, and new roads are not reflected without a data refresh.

No Multi-Waypoint Routing: The system supports only single-pair (origin-destination) routing.

Uniform Cost Model: All roads have equal cost per metre (distance-only). Speed limits, road surface quality, and vehicle type are not considered.


6.3. Future Prospects

The following enhancements are planned or recommended for future development:

Contraction Hierarchies (CH) Integration: Implementing CH preprocessing would reduce query times from seconds to milliseconds, enabling real-time interactive routing even for cross-country distances. This could be achieved by integrating OSRM or GraphHopper as a parallel routing engine.

One-Way Street and Turn Restriction Support: The database setup scripts already include one-way cost assignment logic (reverse_cost = -1 for oneway=yes). Enabling directed := true in the A* query and incorporating turn restriction tables from OSM data would improve route accuracy.

Weighted Cost Functions: Incorporating road type weights, speed limits, and road surface quality into the cost function would produce more realistic travel time estimates, changing the routing objective from shortest distance to shortest time.

Multi-Waypoint and TSP Support: Leveraging pgRouting's pgr_TSP function to support multi-stop route optimisation for logistics and delivery planning.

Vector Tile Rendering: Replacing pre-rendered raster tiles with vector tiles (e.g., using pg_tileserv or Martin) would reduce storage requirements by 10–50x and enable dynamic map styling.

Mobile Deployment: Porting the frontend to a mobile framework (e.g., Ionic/Capacitor or React Native) for tablet-based field deployment.

Alternative Route Computation: Implementing Yen's K-Shortest Paths algorithm (pgr_ksp) to provide users with multiple route alternatives.

Elevation-Aware Routing: Incorporating digital elevation models (DEMs) into the cost function to account for steep gradients that affect heavy vehicle fuel consumption and travel time.

Real-Time Data Overlay: In partially connected environments, supporting optional overlay of real-time data (road closures, weather conditions) when intermittent connectivity is available.

Data Refresh Pipeline: Building an automated pipeline to download, transform, and load updated OSM data into the routing database at scheduled intervals when connectivity is temporarily available.
